## Code-level findings — ranked by (user pain × fix cost)

These are **specific, testable hypotheses** derived from reading `navbar.tsx`, `search-dialog.tsx`, `blog-sidebar.tsx`, `heading-anchor.tsx`, and `pagefind-wrapper.tsx`. Each will be confirmed with a perf trace in Phase 3 before we ship a fix.

### 🔴 CRITICAL — directly match user's reported symptoms

#### 1. Blog sidebar TOC accordion + IntersectionObserver thrash
**File:** `src/components/blog/blog-sidebar.tsx:43-60, 116-137`
**Symptom mapped to:** "anchor texts blinking / jumping around when opening blog pages via deep links"
**Root cause hypothesis:**
- IntersectionObserver uses `rootMargin: "-100px 0px -66%"` and `entries.forEach(e => if (e.isIntersecting) setActiveId(e.target.id))` — if multiple headings intersect during a fast scroll, `activeId` rapidly cycles through them. The loop picks the **last** intersecting entry, not the topmost.
- The H3 child links **only render when the parent H2 is active** (line 116: `{isSectionActive && section.h3s.length > 0 && ...}`) → as `activeId` flickers, the TOC **grows and shrinks**, causing layout shift within the sticky sidebar.
- H3 block has `animate-in slide-in-from-left-1 duration-200` → every re-render re-triggers the CSS animation → visible flicker even if no layout shift.
- `scrollIntoView({ behavior: "smooth" })` on Safari has documented quirks (can cancel, flash to destination, delay oddly).

**Fix plan:**
1. Rank intersecting entries by `intersectionRatio`, pick the one closest to viewport top.
2. Always render H3s (drop the accordion) — only style the active one differently. No layout shift.
3. Drop `animate-in slide-in-from-left-1` on the H3 block.
4. Consider adding `scroll-margin-top: 6rem` to headings so deep links land cleanly below the fixed navbar, reducing the need for smooth-scroll correction.

**Estimated cost:** ~30 LOC, low risk.
**Estimated impact:** directly eliminates the "blinking anchors" symptom.

#### 2. Navbar hydration mismatch + top-banner collapse animation
**File:** `src/components/layout/navbar.tsx:134-150, 194-202`
**Symptom mapped to:** "page jumps around when opening a blog page" (compounds with #1)
**Root cause hypothesis:**
```tsx
const [showTopBanner, setShowTopBanner] = useState(() =>
    typeof window !== "undefined" ? window.scrollY < 50 : true
);
```
- SSR renders with `showTopBanner = true` (top banner visible, `max-h-20`).
- On client hydration of a deep-linked page where `window.scrollY > 50`, initial client state becomes `false` — React swaps the DOM, and the 300ms `transition-all ease-in-out` from `max-h-20 → max-h-0` **animates the collapse visibly** after first paint.
- Everything below the navbar shifts up ~80px during that 300ms transition → visible "jump".

**Fix plan:**
1. On initial mount, skip the transition animation for the first paint (apply a `.no-transition` class that's removed in `requestAnimationFrame` after mount).
2. Or: derive initial state from a CSS-only signal (e.g., `@media` or a root className injected at build time).
3. Or best: replace JS-driven scroll state with CSS scroll-driven animations (`animation-timeline: scroll()`), dropping the scroll listener entirely. Safari 18.4+ supports this — but need to check browser share.

**Estimated cost:** medium, needs careful cross-browser testing.
**Estimated impact:** eliminates the top-banner-induced jump on deep links.

#### 3. Raw scroll listener — per-frame React re-renders
**File:** `src/components/layout/navbar.tsx:141-150`
**Symptom mapped to:** scroll jank on pages with long content
**Root cause hypothesis:**
```tsx
const handleScroll = useCallback(() => {
    const y = window.scrollY;
    setShowTopBanner(y < 50);
    setHasScrolled(y > 10);
}, []);
useEffect(() => {
    window.addEventListener("scroll", handleScroll, { passive: true });
    ...
}, [handleScroll]);
```
- Scroll events fire at 60-120Hz on high-refresh displays.
- Each event calls `setState` twice → React re-renders the whole navbar tree on every frame.
- Safari's React re-render cost is higher than Chromium's.
- `backdrop-blur-xl` (line 209) re-rasters on every paint when `hasScrolled` state toggles.

**Fix plan:**
1. Throttle via `requestAnimationFrame` — only read `scrollY` once per frame.
2. Better: use CSS `scroll-driven animations` or a simple class toggle with `IntersectionObserver` on a sentinel element at the top of the page (no scroll listener at all).
3. Drop `backdrop-blur-xl` on the scrolled state, or reduce to `backdrop-blur-md`.

**Estimated cost:** low (~15 LOC).
**Estimated impact:** measurable INP improvement during scroll, especially on Safari.

#### 4. Pagefind lazy-load on first search click
**File:** `src/components/search/search-dialog.tsx:73-88, 103-131`
**Symptom mapped to:** "search-click jank" — dialog feels slow to open the first time
**Root cause hypothesis:**
```tsx
const loadPagefind = useCallback(async () => {
    if (pagefindRef.current) return;
    const pf = await import(/* webpackIgnore: true */ "/pagefind/pagefind.js");
    ...
}, []);
useEffect(() => {
    if (open) {
        loadPagefind();
        ...
    }
}, [open, loadPagefind]);
```
- First click → dialog opens → `loadPagefind()` begins fetching `/pagefind/pagefind.js` + index shards over cold network.
- Pagefind WASM + shards can be 100-500KB depending on content.
- Meanwhile the dialog renders with no results and no loading indicator for the pagefind init itself.
- Combined with the `setTimeout(() => inputRef.current?.focus(), 50)` (line 114) — focus fires before pagefind is ready, and typing during this window yields "no results" or silent failure.

**Fix plan:**
1. Preload pagefind on **hover** or **focus** of the search button (mouseenter/focusin → start the dynamic import).
2. Or preload on `requestIdleCallback` after main page load.
3. Add a loading state that covers pagefind init, not just search query execution.

**Estimated cost:** low (~15 LOC).
**Estimated impact:** perceived "snappy search" — first click feels instant.

### 🟡 HIGH — worth fixing in the same pass

#### 5. Enable React Compiler
**File:** `next.config.ts` (+ install `babel-plugin-react-compiler`)
**Hypothesis:** React 19.2.1 supports the React Compiler (auto-memoization). Not enabled. The navbar is a large client component with many children; auto-memoization would reduce re-render cost across the whole tree. Direct help for the scroll-listener problem even without fixing #3.
**Cost:** small config change, medium testing risk (need perf bench before/after).
**Impact:** compounds with every other perf fix.

#### 6. `backdrop-blur-xl` cost on Safari
**File:** `navbar.tsx:209`
**Hypothesis:** `backdrop-blur-xl` re-rasters the blur on every paint. Safari's backdrop-filter is particularly expensive. Replacing with `backdrop-blur-md` or dropping entirely would measurably improve scroll paint cost.
**Cost:** 1 LOC.
**Impact:** small-to-medium on Safari scroll jank.

## 🔴 Real measured numbers from Phase 2 harness (added post-harness)

After Phase 2 built the Playwright + web-vitals harness, I ran it against **live shelf.nu** in headless chromium + mobile-safari. Machine-readable dump at `docs/perf-audit/baseline/perf-measurements-2026-04-10.json`.

### The headline findings

| Metric | Browser | Measured | "Good" target | Verdict |
|---|---|---|---|---|
| **Mega menu "Product" open** | Desktop Chromium | **1417 ms** | < 300 ms | 🔴 4.7x over target — confirms menu jank |
| **Search dialog → first results** | Desktop Chromium | **6066 ms** | < 800 ms | 🔴 7.6x over target — pagefind cold load is catastrophic |
| Search dialog open (visible) | Desktop Chromium | 1044 ms | < 200 ms | 🟡 5x over |
| Scroll CLS delta | Desktop Chromium | 0.003 | < 0.05 | ✅ Clean — **but CLS sources trace to `DIV.w-full.transition-all`**, which is literally the top-banner collapse container from navbar.tsx:194-202 — **validates code-read hypothesis #2** |
| Blog post LCP / FCP / TTFB | Desktop Chromium (Webkit same) | 318 / 318 / 126 ms | < 2500 / 1800 / 800 | ✅ Excellent baseline |
| Blog post CLS (initial load) | All browsers | **0** | < 0.1 | ✅ Zero |
| Blog post deep-link CLS delta | All browsers | **0** | < 0.1 | ⚠️ **Does not reproduce the user-reported "blinking anchors" symptom in headless mode** |
| Mobile menu open | iPhone 14 (mobile-safari) | **86 ms** | < 400 ms | ✅ Fast |
| Mobile search first results | iPhone 14 (mobile-safari) | 434 ms | < 800 ms | ✅ Fine |

### Key insights from the real numbers

1. **The jank is desktop-specific.** Mobile Safari is 12-16x faster across the board. Desktop Chromium and desktop WebKit show the same slow numbers, so the fixes will help all desktop browsers. This matches "mostly on Safari" since the user is on a Mac where they use desktop Safari.

2. **Mega menu at 1.4 seconds is the #1 actual pain.** This exceeds my code-reading estimate. Possible causes: 3 large NavigationMenus with 10+ ListItems each = ~30+ Link components re-rendering. Plus the scroll listener re-rendering the navbar tree. Plus no React Compiler memoization.

3. **Search cold load at 6 seconds is catastrophic.** Real users clicking search and waiting 6+ seconds for first results will just leave. This is the single highest-leverage fix in the punch list.

4. **CLS is basically zero everywhere in headless tests.** The "blinking anchors" symptom does **not** reproduce in headless Chromium or WebKit. Three possibilities:
   - The blinking is paint-level flicker (not layout shift), which CLS doesn't measure
   - Headless WebKit differs from real Safari.app on this code path
   - The specific scenarios the tests run don't trigger the bug (e.g., the bug needs a specific scroll position, font load timing, or network race condition)

   **Phase 3 diagnosis is still needed** to see the symptom with real eyes. Options: (a) Playwright MCP interactive session against real Safari.app, (b) user records a screen capture, (c) paint-event tests using PerformanceObserver directly.

5. **The scroll CLS source is exactly the hypothesized element.** `DIV.w-full.transition-all` is the top-banner collapse container in navbar.tsx. The CLS value is tiny (0.003) under test conditions — but on a real deep-link load where the page starts scrolled, the transition would fire and cause a bigger shift. The code-read hypothesis stands.

### What changed in the punch list based on these numbers

- **Fix #4 (search pagefind preload) moves to #1 priority** — 6 seconds → under 1 second is the highest-leverage single change in the entire audit.
- **Fix #3 (scroll listener rAF throttle) merges with "mega menu investigation"** — the scroll listener's re-renders are probably part of why mega menu is slow (the navbar tree keeps re-rendering).
- **Fix #1 (blog sidebar) is still important** but ranked lower because it's not reproducible in headless testing yet — need to see it before shipping a blind fix.
